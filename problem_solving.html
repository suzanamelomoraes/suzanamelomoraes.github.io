A little bit more about problem solving techniques

A common misconception about coders is that their job is to write code, but it fails to capture the essence of what a coder does, which is to solve problems. When confronted with a problem, most coders’ first instinct is to immediately start writing code, but diving into code without a plan is a trap.

Taking the time to understand the problem and structure an effective approach may seem like a time sink, but not only will it lead you to a better solution, it will also save you time in the long run.

The first thing you need to do is to understand the problem. This is crucial. If you don’t understand the parameters of the problem then you can’t possibly expect to develop an effective solution. Can you imagine when you get to the end of the code and realize that it doesn’t address the problem?

Only after that you can start working through potential solutions and write out pseudocode to describe it. Don't forget to anticipate possible issues. If you identify these cases ahead of time then you can plan for them in your solution.

Here are some useful steps to be followed when you need to solve a problem.

1. Understand and Analyse the Problem

Ask yourself:

  • Are you able to understand the question fully?
  • Would you be able to explain the question to another person?
  • What and how many inputs are required?
  •  What would be the output for those inputs
  • Can you separate out some modules or parts from the problem?
  • Do you have enough information? If not, try understanding the question again.

2. Go through the sample inputs and examples thoroughly

Going through some sample inputs and coming up with more examples sure helps you a lot to understand the problem well, and also it gives you a clear way to how many cases your code should handle, and what all can be the possible output or output range.

  • Take very simple examples and find the output
  • Take more complex and bigger inputs to see what will be the output, how many use cases do we want
  • Try out the problem with no input, what should be the output now
  • Try out the problem with invalid input, what should be the output now

3. Break down the problem

  • Divide the problem into subproblems until you get a step you can sort
  • Try to make independent functions for each sub-problem
  • Connect those sub-problems by calling them in the required order, or as necessary (Probably one function would be calling another)

4. Start solving/coding

If you can’t solve the problem fully at once, try writing code for a simpler sub part, as you solve the problem, gradually you keep figuring out the problem in more detail, and you start getting new ideas.

5. Look Back and Learn More

Now, this is the most important part! Once you are done, look back whether the code can be improved, is there any other way to solve the given problem?
You can ask yourself:

  • Is there any other way to solve the problem?
  • Is the code efficient? Can it be more efficient?
  • Is the code readable?
  • If someone else showed you this code, would you be able to understand it?
  • Can the performance be improved?
